DAA CRM Live: From Zero to Cloud Run in One Session

Justin Smith

February 28, 2026 - 3 minutes read - 439 words

Tonight we shipped the DAA CRM from a broken build to a fully deployed, publicly accessible system in a single engineering session. Here’s what that actually looked like.

Starting Point: Five Consecutive Build Failures

The deploy.sh script existed. The Docker images existed. What didn’t exist: a working build.

The first failure was a Go syntax error in backend/main.go — a raw newline embedded inside a string literal:

1
2
3
4
5
6

// Broken
fmt.Printf("DAA CRM Backend starting on port %s...
", port)

// Fixed
fmt.Printf("DAA CRM Backend starting on port %s...\n", port)

Go doesn’t allow unescaped newlines inside string literals. The fix was one character. The lesson: always verify generated code with a local go build before pushing to Cloud Build.

IAM Permission Cascade

With the backend building, the push to GCR failed:

denied: gcr.io repo does not exist. Creating on push requires the
artifactregistry.repositories.createOnPush permission

The Compute service account had artifactregistry.admin but the Cloud Build service account didn’t have roles/editor. GCR on newer GCP projects is backed by Artifact Registry — and it needs explicit creation permissions. We granted roles/editor to both SAs:

1
2
3

gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:87748455115@cloudbuild.gserviceaccount.com" \
  --role="roles/editor"

The Missing Frontend Scaffold

The React frontend had src/App.tsx, src/EnrollmentForm.tsx, and src/types.ts — but no public/index.html, no src/index.tsx, and an invalid tsconfig.json. react-scripts build requires all three.

We created the missing files and caught a subtle TypeScript error: inside the if (view === 'enroll') early-return block, TypeScript’s control flow narrowed view to the literal type 'enroll', making view === 'admin' a TS2367 dead comparison. The fix was refactoring to a single return with conditional rendering using view as string.

Replacing Terraform with Direct Deployment

The deploy.sh script ended with:

1
2

cd ../../infra/terraform
terraform init && terraform apply -auto-approve

Terraform wasn’t installed. The infra directory didn’t exist. Rather than block on tooling, we replaced the step with direct gcloud run deploy calls — both backend and frontend were live within 60 seconds.

What’s Running Now

ServiceURL
Backend API (Go)daa-crm-backend-87748455115.us-central1.run.app
Registration Portal (React)daa-crm-frontend-87748455115.us-central1.run.app

The CRM is multi-tenant, supporting both DAA-CORE and BGC-METRO environments via a tenant selector. Students submit inquiries through the public portal; staff manage records through the admin dashboard within the same app.

What’s Still Missing

The backend stores student data in memory. A cold start (which Cloud Run performs after idle periods) wipes all records. Connecting to Cloud Firestore or Cloud SQL is the immediate next priority before the March 8 cohort start.
