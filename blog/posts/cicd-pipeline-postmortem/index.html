Rearchitecting the Automated Insights CI/CD Pipeline: What Kept Breaking and Why

Justin Smith

February 28, 2026 - 3 minutes read - 529 words

The Automated Insights blog has had a recurring CI/CD problem: deployments that work locally fail silently in GitHub Actions, or succeed in Actions but produce a broken site. Here’s the root cause analysis and what we changed.

The Failure Pattern

The pipeline kept failing in one of three modes:

  1. Hugo build succeeds, site content missing — the public/ directory was generated but the git commit step failed silently because there was nothing new to commit (empty diff). The deploy step ran anyway and pushed a stale build.

  2. Theme submodule not initialized — the Ananke theme is a git submodule. Fresh GitHub Actions runners clone the repo but don’t automatically initialize submodules. The build proceeded with an empty themes/ananke/ directory.

  3. Cache invalidation lag — CDN-cached pages were served stale after a new deployment, making it appear the deployment failed when it had actually succeeded.

Root Cause: Submodule Initialization

The most persistent failure was the submodule issue. The fix is adding a single flag to the checkout step in the workflow:

2
3
4

- uses: actions/checkout@v4
  with:
    submodules: true        # ← this was missing
    fetch-depth: 0          # needed for git info in Hugo

Without submodules: true, themes/ananke/ is present as an empty directory. Hugo finds the theme = "ananke" setting, looks in themes/ananke/layouts/, finds nothing, and falls back to default templates — producing a completely unstyled build that nonetheless exits with code 0.

The Empty Commit Problem

GitHub Actions was pushing even when no content changed, because the commit-and-push step wasn’t checking for an actual diff:

# Before — always commits, fails when nothing changed
- run: |
    git add public/
    git commit -m "Deploy"
    git push

# After — skips if nothing changed
- run: |
    git add public/
    git diff --staged --quiet || git commit -m "chore: deploy $(date -u +%Y-%m-%dT%H:%M:%SZ)"
    git push

The || git commit pattern: if git diff --staged --quiet exits 0 (no changes), skip the commit. If it exits non-zero (changes present), run the commit.

The Clawd Agent Angle

Because the Automated Technologies enterprise runs on the Clawd autonomous agent framework, we also set up a qa_gate.py agent to verify deployments. The QA gate:

  1. Fetches the live site URL after a deploy
  2. Checks for the presence of the latest post title in the HTML
  3. Verifies the Last-Modified response header is newer than the previous run’s timestamp
  4. Fails the pipeline if either check doesn’t pass

This turns deployment verification from a manual step into an automated gate — consistent with how we want all agents in the framework to operate.

Current State

The pipeline now runs reliably. Outstanding items:

  • Multilingual posts — German and Russian translations are still sparse. The English content is authoritative; translations are queued for Sprint 3 of the blog roadmap.
  • Search — Ananke doesn’t include built-in search. We’re evaluating Pagefind (client-side) as a zero-dependency solution.
  • Analytics — Currently none. Privacy-first means no tracking pixels, but we’ll add a self-hosted Plausible instance when the infrastructure plan is finalized.

Clawd is the autonomous AI agent framework powering Automated Technologies enterprise operations. The qa_gate.py agent is part of the Clawd agents directory.
