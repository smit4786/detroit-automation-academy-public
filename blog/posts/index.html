@AT_Diary Sprint 2 Complete: On-Device Intelligence with Apple Frameworks

Sprint 2 of @AT_Diary is complete — build successful. Here’s what we shipped.

What Is @AT_Diary?

@AT_Diary is a privacy-first iOS journaling application that automatically synthesizes your daily life into a structured diary. No servers. No cloud storage. No data leaves your device.

The application pulls context from four native Apple frameworks and weaves them into a coherent daily narrative — entirely on-device.

What We Built in Sprint 2

Apple Framework Integration

FrameworkData SynthesizedStatus
MusicKitWhat you listened to, when, how long✅ Sprint 2
EventKitCalendar events attended, scheduled meetings✅ Sprint 2
CoreLocationSignificant locations visited throughout the day✅ Sprint 2
PhotosPhotos captured during the day (metadata only, on-device)✅ Sprint 2

Key Technical Decisions

Why native frameworks only?
The moment data touches a network, you need a privacy policy, a server to secure, and a trust relationship with the user that’s easy to destroy. Native-only means the diary is as private as the iPhone itself.

DAA CRM Live: From Zero to Cloud Run in One Session

Tonight we shipped the DAA CRM from a broken build to a fully deployed, publicly accessible system in a single engineering session. Here’s what that actually looked like.

Starting Point: Five Consecutive Build Failures

The deploy.sh script existed. The Docker images existed. What didn’t exist: a working build.

The first failure was a Go syntax error in backend/main.go — a raw newline embedded inside a string literal:

1
2
3
4
5
6

// Broken
fmt.Printf("DAA CRM Backend starting on port %s...
", port)

// Fixed
fmt.Printf("DAA CRM Backend starting on port %s...\n", port)

Go doesn’t allow unescaped newlines inside string literals. The fix was one character. The lesson: always verify generated code with a local go build before pushing to Cloud Build.
